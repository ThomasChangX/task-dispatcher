# 6 阶段工作流详细说明

## 流程概览

```
需求分析 → 研究分析 → 方案确认 → 任务拆解 → 执行验证 → 收尾
    ↓          ↓          ↓           ↓          ↓         ↓
  目标提取   信息收集   等确认     分发Agent   验证质量   归档记录
```

---

## 阶段 1：需求分析

### 执行时机
用户提交任何任务时立即触发

### 核心动作
1. **提取任务目标**：用户想要达成什么
2. **识别约束条件**：时间、预算、技术限制
3. **判定任务类型**：修复/新功能/调研/文档/架构/部署
4. **评估复杂度**：简单/一般/复杂/超大

### 任务类型判定规则

| 关键词/特征 | 任务类型 |
|-------------|----------|
| "bug"、"修复"、"报错"、"崩溃" | 修复 |
| "新增"、"添加"、"实现"、"开发" | 新功能开发 |
| "调研"、"调查"、"了解"、"研究" | 调研 |
| "文档"、"说明"、"教程" | 文档任务 |
| "设计"、"架构"、"方案" | 架构设计 |
| "部署"、"上线"、"运维" | 部署运维 |

### 复杂度评估

| 级别 | 特征 | 预估时间 |
|------|------|----------|
| 简单 | 单文件修改，逻辑清晰 | <30min |
| 一般 | 多文件修改，需测试 | 30min-2h |
| 复杂 | 跨模块改动，需设计评审 | 2h-8h |
| 超大 | 多模块重构，新系统 | >8h |

### 产出格式

```markdown
## 任务分析报告

### 目标
[用户想要达成什么]

### 约束条件
- [条件1]
- [条件2]

### 任务类型
[判定类型]

### 复杂度
[简单/一般/复杂/超大]
```

---

## 阶段 2：研究分析

### 执行时机
需求分析完成后

### 核心原则
**必须执行，不可跳过！** 没有研究分析的方案是不负责任的。

### 研究内容

| 任务类型 | 研究重点 |
|----------|----------|
| 修复 | 问题根因、相关代码、类似问题 |
| 新功能 | 技术方案、依赖关系、最佳实践 |
| 调研 | 背景信息、现有方案、优缺点对比 |
| 文档 | 参考文档、现有内容、结构 |
| 架构 | 技术选型、风险评估、可行性 |
| 部署 | 环境要求、配置项、流程 |

### Agent 调用
调用 **researcher** agent 执行研究分析

### 产出格式

```markdown
## 研究报告

### 背景
[相关背景信息]

### 发现
- [发现1]
- [发现2]

### 建议方案
[基于研究的建议]

### 风险点
- [风险1]
- [风险2]

### must_haves
> 可验证的硬性交付标准，作为 prompt 独立章节嵌入后续执行阶段
- [标准1]
- [标准2]
```

---

## 阶段 3：方案确认

### 执行时机
研究分析完成后

### 核心动作
1. 整理方案要点
2. **调用 critic 进行方案评审**
3. 展示方案给用户
4. **等待确认后才继续**

### 审核节点：方案评审（critic）

**触发条件**：must_haves 已明确且可验证

| 项目 | 说明 |
|------|------|
| 执行者 | critic |
| 评估内容 | 技术方案、风险、must_haves 完整性 |
| 通过标准 | 方案可行 + must_haves 明确可验证 |

### 审核流程

```
研究分析完成
    ↓
[检查 must_haves 是否明确]
    ↓
┌─ 是 → 调用 critic 方案评审
│        ↓
│    [评审通过] → 进入任务拆解
│    [评审失败] → 重新进入研究分析
└─ 否 → 先补充 must_haves
```

### 确认内容

| 项目 | 说明 |
|------|------|
| 任务目标 | 是否正确理解 |
| 研究结论 | 是否认可 |
| 方案思路 | 是否同意 |
| must_haves | 是否清晰可验证 |
| 预期产出 | 是否满足需求 |

### 判定规则

- 审核通过 + 用户同意 → 进入任务拆解
- 审核失败 → 重新进入研究分析阶段
- 用户有异议 → 记录异议，重新分析
- 用户超时未回复 → 提醒一次，继续等待

---

## 阶段 4：任务拆解

### 执行时机
方案确认通过后

### 核心动作
1. 将任务拆解为可执行的子任务
2. 确定子任务的执行顺序（串行/并行）
3. 为每个子任务分配合适的 agent
4. 生成任务表格

### 任务表格格式

| 序号 | 任务 | Agent | must_haves | 依赖 | 状态 |
|------|------|-------|------------|------|------|
| 1 | xxx | coder | - | - | 待执行 |
| 2 | xxx | tester | 验证点 | 1 | 待执行 |
| 3 | xxx | reviewer | 审查标准 | 2 | 待执行 |

### 拆解原则

- 每个子任务必须有明确产出
- 明确任务间的依赖关系
- 并行任务需说明可并行执行
- 预留评审节点

---

## 阶段 5：执行验证

### 执行时机
任务拆解完成后

### 核心动作
1. 按依赖顺序分发 sub-agent
2. 实时跟踪执行状态
3. 每个关键节点汇报进度
4. **调用审核节点验证产出**
5. **循环检测检查点**（P0）

### 审核节点：执行验证

#### 审核 1：结果评估（reviewer）
**触发条件**：子任务完成后

| 项目 | 说明 |
|------|------|
| 执行者 | reviewer |
| 评估内容 | 输出质量、完整性 |
| 通过标准 | 产出符合 must_haves |

#### 审核 2：测试验证（tester）
**触发条件**：功能实现完成后

| 项目 | 说明 |
|------|------|
| 执行者 | tester |
| 评估内容 | 功能正确性 |
| 通过标准 | 功能符合 must_haves |

### 审核触发规则

```
子任务执行完成
    ↓
[检查 must_haves 是否满足]
    ↓
┌─ 是 → 调用 reviewer 进行结果评估
│        ↓
│    [评估通过] → 调用 tester 进行测试验证（如适用）
│    [评估失败] → 返回修复 → 重新执行
│                  ↓
│            [修复后重新审核]
│                  ↓
│            [连续失败3次] → 重新进入研究分析阶段
│
└─ 否 → 标记未满足 → 继续执行 → 最后统一审核
```

### 核心规则

1. **审核必须全部通过** - 所有审核节点必须通过才算完成任务
2. **审核不通过 → 重新进入研究分析阶段**
3. **must_haves 必须完成才能通过审核**

### 循环检测检查点（P0）

每次子任务执行前调用 LoopDetector 检查：

```python
from scripts.loop_detector import get_detector

detector = get_detector()
result = detector.check(task_id)

if not result['safe']:
    if result['action'] == 'alert':
        # 达到阈值，触发告警
        logger.warning(f"任务 {task_id} 已执行 {result['count']} 次，达到阈值")
    elif result['action'] == 'fallback':
        # 超过阈值，进入兜底
        logger.error(f"任务 {task_id} 执行次数过多，进入兜底处理")
        # 记录兜底原因，返回用户
```

#### 检查点规则

设计规则: **≤3 次安全，>3 次兜底**

| 执行次数 | safe | action | 后续动作 |
|----------|------|--------|----------|
| 0 | true | continue | 正常执行 |
| 1 | true | continue | 正常执行 |
| 2 | true | continue | 正常执行 |
| 3 | true | continue | 正常执行 |
| >3 | false | fallback | 兜底处理 |

### 评估节点

| 节点 | 执行者 | 评估内容 |
|------|--------|----------|
| 代码完成后 | reviewer | 代码规范、安全检查 |
| 功能完成后 | tester | 功能正确性、覆盖率 |
| 方案完成后 | critic | 风险评估、可行性 |

### must_haves 对照检查
执行完成后，逐条对照检查 must_haves 是否满足：

```markdown
## must_haves 验证

| 标准 | 状态 | 验证结果 |
|------|------|----------|
| 标准1 | ✅/❌ | [验证详情] |
| 标准2 | ✅/❌ | [验证详情] |

- 未满足项: [如有]
- 处理方案: [如有]
```

### 失败处理

- 单个子任务失败 → 分析原因 → 重新生成计划 → 评审 → 继续
- 连续失败 → 兜底记录 → 汇报用户

---

## 阶段 6：收尾

### 执行时机
所有子任务验证通过后

### 核心动作
1. 汇总执行结果
2. 更新 SESSION-STATE.md 为完成
3. 记录到 memory/YYYY-MM-DD.md
4. 更新 MEMORY.md（如有重要经验）
5. 向用户汇报完成

### 记录模板

```markdown
## 任务完成记录

### 任务
[任务描述]

### 完成时间
[ISO 时间]

### 产出
- [产出1]
- [产出2]

### 经验教训
- [如有]
```

---

## 决策点汇总

| 节点 | 决策 | 后续动作 |
|------|------|----------|
| 需求分析后 | 任务类型/复杂度 | 进入研究分析 |
| 研究分析后 | must_haves 是否明确 | 是 → 进入方案确认 / 否 → 补充 must_haves |
| 方案确认前 | critic 方案评审是否通过 | 通过 → 等待用户确认 / 失败 → 重新进入研究分析 |
| 方案确认后 | 用户是否同意 | 进入任务拆解 / 重新分析 |
| 任务执行中 | 是否遇到问题 | 修复后继续 / 兜底处理 |
| 任务执行后 | reviewer/tester 审核是否通过 | 通过 → 进入收尾 / 失败 → 重新进入研究分析 |
| 任务完成后 | 是否有经验值得记录 | 更新记忆系统 |

---

## 审核汇总规则

### 核心规则

1. **审核必须全部通过** - 所有审核节点（方案评审、结果评估、测试验证）必须通过
2. **审核不通过 → 重新进入研究分析阶段**
3. **must_haves 必须完成才能通过审核**

### 审核节点清单

| 节点 | 执行者 | 触发阶段 | 评估内容 | 通过标准 |
|------|--------|----------|----------|----------|
| 方案评审 | critic | 方案确认 | 技术方案、风险、must_haves | 方案可行 + must_haves 明确 |
| 结果评估 | reviewer | 执行验证 | 输出质量、完整性 | 产出符合 must_haves |
| 测试验证 | tester | 执行验证 | 功能正确性 | 功能符合 must_haves |

### 审核失败流程

```
审核失败
    ↓
[记录失败原因]
    ↓
[分析失败原因]
    ↓
┌─ 可修复 → 修复后重新审核
│          ↓
│    [连续失败3次] → 重新进入研究分析阶段
│
└─ 不可修复 → 重新进入研究分析阶段
```
